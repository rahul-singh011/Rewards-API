# Rewards & Ranking API

A points-based rewards system with idempotent transactions, concurrency-safe writes, and multi-factor ranking. Built with Python (FastAPI) + SQLite for the backend and React (Vite) for the frontend.


## API Reference

### POST /transaction

Records an earn or spend transaction for a user.

**Request body:**
```json
{
  "user_id": "user_001",
  "amount": 150.0,
  "type": "earn",
  "idempotency_key": "unique-uuid-here"
}
```

**Rules:**
- `type` must be `"earn"` or `"spend"`
- `amount` must be greater than 0
- `idempotency_key` must be unique per request — reuse the same key to safely retry without side effects
- Cannot spend more than current balance
- Max 10 transactions per user per hour

**Responses:**

| Status | Meaning |
|--------|---------|
| 201 | Transaction created successfully |
| 400 | Invalid input or insufficient balance |
| 404 | User not found |
| 422 | Validation error (wrong type, missing field) |
| 429 | Rate limit exceeded |

**Example response:**
```json
{
  "transaction_id": "6694de589043356940b4d0036e2ea957",
  "user_id": "user_001",
  "amount": 150.0,
  "type": "earn",
  "new_balance": 150.0,
  "message": "Successfully earned 150.0 points"
}
```

```bash

```

**Example response:**
```json
{
  "user_id": "user_001",
  "username": "alice",
  "balance": 100.0,
  "total_earned": 150.0,
  "total_spent": 50.0,

```

| Status | Meaning |
|--------|---------|
| 200 | User found, data returned |
| 404 | User not found |

### GET /ranking

Returns all users sorted by composite ranking score.

**Example:**
```bash
curl http://localhost:8000/ranking
```

**Example response:**
```json
{
  "users": [
    {
      "rank": 1,
      "user_id": "user_001",
      "username": "alice",
      "score": 1.0,
      "balance": 100.0,
      "total_earned": 150.0,
      "txn_count": 2,
      "last_active_at": "2026-06-24 11:04:20"
    }
  ],
  "scoring_formula": "score = (normalized_total_earned × 0.5) + (normalized_txn_count × 0.3) + (recency_score × 0.2)"
}
```

---

## How Ranking is Calculated

Ranking uses a **3-factor weighted composite score**:

score = (normalized_total_earned × 0.5)

+ (normalized_txn_count    × 0.3)

+ (recency_score           × 0.2)

| Factor | Weight | Why |
|--------|--------|-----|
| Total points earned | 50% | Core measure of activity volume |
| Transaction count | 30% | Rewards consistent engagement |
| Recency | 20% | Penalizes long-inactive users |

**Normalization:**
Each factor is min-max normalized across all users to `[0, 1]` before combining. This prevents large raw numbers (e.g. thousands of points) from drowning out smaller factors (e.g. transaction count in the tens).

**Recency score formula:**
recency_score = 1 / (1 + days_since_last_active)

- Active today → ~1.0
- Active 1 day ago → 0.5
- Active 7 days ago → 0.125
- Active 30 days ago → 0.032

This is a smooth decay — no hard cutoffs that could be gamed.

## How Duplicate Requests Are Prevented

Every `POST /transaction` request must include an `idempotency_key` (a UUID generated by the client).

**Flow:**
1. On every request, we check the `idempotency_keys` table first
2. If the key exists → return the stored response immediately, no writes happen
3. If the key is new → process the transaction, then store the key + response
4. If two identical requests arrive simultaneously → `UNIQUE` constraint on the DB catches the second one and returns the stored response

This means it is always safe to retry a failed request with the same key — you will never be charged or credited twice.

---

## How Concurrency is Handled

Two layers of protection:

**Layer 1 — App level:**
A `threading.Lock()` (`write_lock`) ensures only one write operation runs at a time inside the Python process. Any concurrent request waits until the lock is released.

**Layer 2 — Database level:**
`BEGIN IMMEDIATE` tells SQLite to acquire a write lock at the DB level. No other writer can start until the current transaction commits or rolls back.

**Double-checked locking:**
Balance is validated once before acquiring the lock (fast path) and again inside the lock after re-reading from DB (safe path). This handles the race condition where two spend requests both pass the balance check simultaneously but only one should succeed.
